## 二、我对无损压缩任务的理解

做这个网站的过程中，我也重新梳理了自己对无损压缩任务的理解。以前如果很粗地说，可能会说无损压缩就是“把文件变小”。但我现在觉得这个说法只说到了表面。

我现在的理解是，无损压缩的本质是在保证完全可逆还原的前提下，用更少的 bit 表示原始数据。这里最重要的约束是“完全可逆”。解压之后的文件必须和原始文件逐字节一致，最好用 SHA-256 这样的校验做闭环。只要解压结果不一致，不管视觉上多接近，或者语义上多相似，都不能叫严格无损压缩。

从这个角度看，压缩器真正压掉的不是“文件大小”，而是数据里的冗余。冗余可以有几类：符号频率冗余，比如高频符号应该用更短码；重复片段冗余，比如前面出现过的片段后面又出现；上下文规律冗余，比如看到前几个字节后可以预测下一个字节；结构化数据冗余，比如图像相邻像素相似、程序代码有语法模式、医学图像有头部和像素区、科学数组有局部连续性。

这里我觉得最关键的是，压缩率的上限和数据本身的信息量、熵以及可预测性有关。不是所有数据都能压得很小。如果一个文件接近随机，或者已经被强压缩、加密过，通用压缩器再压一次通常不会有明显收益，甚至可能因为头部和封装变大。反过来，如果数据里有大量重复、强上下文规律或者领域结构，压缩器就能利用这些规律。

我还特别修正了一个表达：不能说“压缩器输入一般是字符”。更准确地说，压缩器输入是符号序列，最常见的是 bit 或 byte 序列。字符只是文本文件在人眼里的表现，进入压缩器之后通常还是编码后的字节流。比如中文文本在人眼里是汉字，但压缩器看到的是 UTF-8 字节；医学图像在人眼里是图像切片，但通用压缩器看到的是 byte stream；Silesia 中的文本、程序、图像和医学数据，本质上都可以进入通用压缩器，只是不同数据能利用的冗余不同。

所以我现在看无损压缩时，会把“预测”看成核心。一个模型越能准确预测下一个符号，理论上就越能用更少 bit 表示它。概率越高，理想码长越短；概率越低，码长越长。从 Huffman 到 PPM、PAQ，再到神经网络压缩，很多方法其实都在回答同一个问题：怎样更准确地估计符号概率，并把这个概率变成更短的 bitstream。

## 三、数据类型与冗余特点

我现在还有一个认识：无损压缩不能只看算法，也要看数据。不同数据压缩难度不同，不是因为某个压缩器名字更高级，而是因为数据本身的冗余结构不同。

文本数据通常有词频、语言上下文和重复短语。自然语言里常用词、空格、标点、固定表达会反复出现；文档里还会有标签、模板、格式符号。程序代码虽然也是文本，但它的结构更强，有关键字重复、语法结构、缩进、函数名、变量名、导入语句和模板代码。这类数据适合用字典匹配、BWT、PPM、PAQ、Brotli 等方法利用重复和上下文。

图像数据的冗余更多来自空间相关性。相邻像素通常相似，颜色通道之间可能相关，所以图像压缩经常先做预测、差分、滤波或颜色变换，再对残差编码。PNG filter、JPEG-LS 的预测残差、WebP lossless 的变换，本质上都是先把图像变成更容易被后端压缩的形式。

医学图像和科学数据又更复杂。医学图像可能有文件头、元数据、切片结构、不同位深和像素区；科学数组可能有浮点连续性、局部平滑性、高位结构和低位噪声。对这类数据，通用压缩器可以作为 baseline，但如果要压得更好，往往需要领域预处理，比如 delta、bit-plane、shuffle、residual 或者专用格式。

这也自然引出 Silesia corpus。我选择 Silesia 的原因是，它不是单一文本数据，而是混合数据集，包含多种类型，比较适合做通用压缩器 baseline。主要也是甲方要求关注 Silesia。它能帮助我观察 gzip、zstd、xz、PAQ8PX、cmix 这类压缩器在混合数据上的表现差异。不过我也会强调，Silesia 是基准，不是最终业务结论。如果甲方业务数据分布和 Silesia 差异很大，后续还要用真实业务样本或脱敏数据补测。

## 四、压缩器核心框架

我现在看一个压缩器，不再先问它叫什么名字，而是先问它由哪些核心部分组成。对我来说，可以拆成三个问题。

第一个问题是数据表示。输入到底是 bit、byte、字符、token、block，还是领域对象？多数通用压缩器面对的是字节流，PAQ 系统经常走 bit-level prediction，神经压缩可能处理 byte、token 或 latent，图像和科学数据压缩则可能先转换成残差、bit-plane 或 block。输入表示不同，后面的建模方式也不同。

第二个问题是冗余发现或概率建模。它怎么发现重复？怎么利用上下文？怎么预测下一个符号？LZ77 主要通过滑动窗口找重复片段；BWT 先重排数据，让相似上下文聚集；PPM 根据上下文估计下一个符号；PAQ/CMIX 用大量上下文专家和 mixer 合成概率；神经压缩用 RNN、Transformer 或其他可学习模型做概率预测。

第三个问题是熵编码。模型给出符号概率之后，还要把概率落成 bitstream。Huffman 编码简单、成熟、适合整数位长前缀码；Arithmetic Coding 和 Range Coding 更接近熵下界，适合强概率模型；ANS/rANS/tANS 兼顾接近算术编码的效率和较高工程吞吐，在现代系统和学习式压缩里很重要。

所以我会把压缩器逻辑链写成：

原始数据 -> 符号序列/字节流 -> 冗余发现/概率建模 -> 熵编码 -> 压缩文件 -> 解压端完全还原。

不同压缩器的差异，本质上主要体现在四件事上：它发现冗余的能力、概率预测能力、熵编码方式，以及速度、内存、兼容性这些工程权衡。gzip 成熟兼容但建模能力有限；zstd 在速度和压缩率之间做了工程平衡；xz/LZMA 压缩率更高但压缩更慢；PAQ8PX/cmix 预测能力强但速度和内存代价高；神经网络压缩可能预测更强，但模型和部署成本更高。

这个框架也能自然过渡到算法演进。因为每一代压缩方法，其实都在围绕“怎样更好发现冗余、怎样更好估计概率、怎样更高效写成 bitstream”推进。

## 五、算法演进谱系

我现在不想把算法演进讲成百科，而是把它理解成一条问题逐步深化的线。

Huffman 从频率统计出发。它回答的是：如果符号频率已知，怎样给高频符号短码、低频符号长码，从而降低平均码长。它解决的是已知概率后的编码问题，是理解熵编码最基础的入口。

LZ77/LZ78 把注意力从单个符号扩展到重复片段。LZ77 的滑动窗口思想很重要：前面出现过的内容，如果后面又出现，就用距离和长度这样的引用表示。LZ78 则走动态短语字典路线。它们让通用压缩器开始真正利用序列中的重复结构。

DEFLATE/gzip 是工程组合的典型例子。DEFLATE 不是一个孤立算法，而是 LZ77 匹配加 Huffman 编码。先找重复，形成 literal、length、distance 这些 token，再对 token 做 Huffman 编码。gzip 则是常见封装和工具生态。它适合作为经典兼容 baseline，不是因为它最强，而是因为它稳定、成熟、可复现、容易作为兜底参照。

BWT/bzip2 代表“先变换，再压缩”的思路。BWT 本身不是完整压缩器，而是通过可逆重排让相似上下文聚集，再配合 MTF、RLE、Huffman 等步骤。它给我的启发是，有时不是后端编码器不够强，而是原始数据还没有变成容易预测的形式。

PPM 开始强调上下文概率建模。它根据前面的上下文预测下一个符号，而不是只看全局频率。这个思路对文本压缩很重要，也自然通向 PAQ 和 cmix。

PAQ、PAQ8PX 和 cmix 是高压缩率路线。它们通过大量上下文模型、模型混合、概率更新来提升预测精度，再接 Range Coding 或 Arithmetic Coding 输出 bitstream。我现在把 PAQ8PX 定位成后续重点拆解对象，因为它能代表传统高压缩率压缩器到底怎么利用上下文、预处理和模型混合。

zstd、LZMA/xz 和 brotli 是现代工程压缩器。它们不只追求压缩率，还要考虑速度、内存、兼容性和部署生态。zstd 适合做现代高速工程 baseline，LZMA/xz 适合做高压缩率工程 baseline，brotli 适合 Web 文本场景参考。

神经网络压缩则可以理解为用可学习模型做更强概率预测。NNCP、DeepZip、TRACE、Language Modeling Is Compression 这类方法，本质上还是预测下一个符号，只是预测器从手工上下文模型变成了 RNN、Transformer 或大语言模型。但它们也带来速度、模型大小、显存、训练成本和部署复杂度问题，所以当前更适合作为前瞻研究方向，而不是默认业务方案。

如果用一句话概括这条演进线，我会说：

频率统计 -> 重复匹配 -> 数据变换 -> 上下文预测 -> 模型混合 -> 神经网络概率建模。

这不是说后者完全替代前者，而是说压缩器越来越强调对数据规律的建模能力。模型越强，通常压缩潜力越大，但代价也越高，所以必须进入实验评估。

## 六、代表压缩器精读和复现选择

后面我不会盲目把所有压缩器都复现一遍，而是先建立一组代表性 baseline 和精读路线。

第一类是 gzip / DEFLATE。它是经典兼容 baseline，帮助我理解 LZ77 + Huffman 的基础结构。它也是后续所有实验里最基础的参照：任何方法都要能说清楚相对 gzip 多省了多少，又多付出了多少时间和资源。

第二类是 zstd 和 LZMA/xz。zstd 代表现代高速工程压缩器，重点看速度、压缩率、内存和部署之间的 trade-off。LZMA/xz 代表高压缩率工程路线，适合离线归档和分发场景。它们能帮助我建立工程 baseline，而不是只盯着极限压缩率。

第三类是 PAQ8PX。它是高压缩率传统压缩器的重点拆解对象。我后续想重点看它的上下文模型、模型混合、概率更新、文件类型检测、预处理和 Range Coding。这里我不是想直接把 PAQ8PX 搬到业务里，而是要理解它强在哪里，哪些思想可以抽象成后续自研模块。

扩展调研对象包括 cmix、NNCP、DeepZip 和 Language Modeling Is Compression。cmix 用来和 PAQ8PX 对比极限 context mixing 路线；NNCP 用来理解神经网络通用无损压缩；DeepZip 更适合理解 RNN predictor + Arithmetic Coding 这个基本组合；Language Modeling Is Compression 则说明强语言模型和无损压缩之间的关系。

我会明确说，这些不是都已经复现完成了。现在是建立重点精读和复现路线，避免盲目看论文。真正复现时，我会按数据集、压缩器版本、参数、硬件、计时方式和 bit-exact 校验统一口径。

## 七、Silesia 实验展示

Silesia 实验这一部分，我现在会定位成实验设计和展示框架，而不是已经完成的本地实测结果。当前项目里已经有 Experiments 页面、数据集选项、baseline 选项和指标定义，但我还没有把 Silesia 全 12 个文件在同一环境下完整跑完，所以不能编造结果。

这个实验的目标不是证明我超过了谁，也不是刷榜，而是建立可复现 baseline。具体目标有三个：在统一数据集上测试不同压缩器；观察不同技术路线的 trade-off；形成后续改进算法的评价基准。

实验对象我会按技术路线分层。必选是 gzip、zstd、xz/LZMA：gzip 代表经典兼容基线，zstd 代表现代速度和综合性能，xz/LZMA 代表高压缩率工程路线。可选是 brotli、PAQ8PX、cmix、NNCP/DeepZip：brotli 对应 Web 文本场景，PAQ8PX 是甲方指定关注对象，也是公开高压缩率路线代表，cmix 用来对比极限 context mixing，NNCP/DeepZip 用来跟踪神经概率建模方向。

指标我会分三层。

第一层是基础指标：原始大小、压缩后大小、compression ratio、BPB、压缩时间、解压时间、内存峰值、SHA-256 round-trip 一致性。这里 BPB 很重要，因为甲方技术评审更习惯看 bits per byte，而不是只看 ratio。bit-exact 是底线，不一致直接判定失败。

第二层是衍生指标：压缩吞吐 MB/s、解压吞吐 MB/s、BPB 相对 gzip 的节省比例、解压/压缩时间比、内存字节/原始字节，以及效费比指标，也就是 ratio 或 BPB 每提升一点要付出的时间代价。

第三层是场景指标：如果甲方关注小包或在线服务，就补 32KB、64KB、128KB 块大小下的 P50、P95、P99 和 max 延迟；如果关注冷热数据，就分开测；如果关注服务吞吐，就看多线程吞吐。

实验口径上，我会统一 Silesia 数据集、硬件、压缩器版本、编译方式、参数和计时方式。每个文件都记录原始 SHA-256 和解压后 SHA-256，所有命令和版本号进入附录。时间指标最好多次运行取中位数，避免单次波动。

后续真正有结果时，我会按“三句话框架”讲每个压缩器：先一句话定位技术路线，再用一个 ratio/BPB 和 speed 点讲 trade-off，最后给场景建议。比如 gzip 适合通用低开销，zstd 适合高 QPS 后端，xz 适合归档分发，PAQ8PX/cmix 适合离线极限压缩和研究对比但不适合业务在线部署，NNCP/DeepZip 作为前瞻方向跟踪。

## 八、多维评估

我现在评价一个压缩器，不会只看压缩率。尤其面对甲方或真实业务，最重要的问题是：这个东西能不能用，用在什么地方，值不值得投入。

第一是压缩率。我会看压缩后绝对大小、BPB，以及不同文件类型上的稳定性。稳定性很重要，因为甲方数据通常不是单一文本，而是混合类型。某个算法在一类文件上特别好，但在另一类文件上崩掉，在业务里就是风险。

第二是速度。压缩时间和解压时间要分开看。归档场景可以接受慢压缩，但在线服务、API 网关、边缘节点、小包传输不能接受秒级延迟。解压速度尤其重要，因为很多系统是一次压缩，多次读取或分发。

第三是资源开销。要看内存峰值、CPU、是否依赖 GPU、是否能多线程扩展。PAQ8PX/cmix 如果内存和时间成本太高，就更适合研究和离线极限压缩。NNCP/DeepZip 如果依赖 GPU 或大模型，就要把显存、模型管理、推理成本和训练成本算进去。

第四是兼容性和部署成本。gzip 和 zstd 的优势是成熟、跨平台、库和 CLI 都容易集成。xz 也适合归档生态。PAQ8PX/cmix 部署成本更高，需要单独编译、维护和排错。神经压缩更像独立项目，而不是简单引入一个库。

第五是研究价值。有些压缩器不适合部署，但很值得拆解。比如 PAQ8PX/cmix 的上下文模型、模型混合、文件类型路由、概率更新机制，都可能抽象成自研模块。神经压缩虽然工程化难，但它提供了新的概率建模思路，可以作为后续增强传统上下文模型的切入点。

基于这五个维度，我目前的阶段性判断是：

gzip 是经典兼容 baseline。压缩率不是最强，但成熟、稳定、部署成本低，适合作为兜底对照。

zstd 是现代高速工程 baseline。在速度、压缩率、内存和生态之间平衡得好，如果没有极端压缩率要求，通常是工程首选。

LZMA/xz 是高压缩率工程 baseline。适合离线归档、分发包和不频繁解压的数据。

PAQ8PX/cmix 是高压缩率研究对象。它们适合研究对标和极限离线压缩，不适合默认业务部署。

NNCP、DeepZip、Language Modeling Is Compression 是神经网络概率建模方向。它们代表下一代压缩器可能的研究路线，但需要单独评估模型大小、速度、资源和工程化风险。

如果面向甲方，我最终会给一个简单建议框架：先按场景选路线，再按数据选档位。通用在线场景走 zstd，归档场景走 xz，极限压缩场景用 PAQ8PX/cmix 做离线对标但不默认上线，神经网络方向单独立项跟踪。

## 九、后续研发方向

后续我不想把计划写成“继续学习”，而是进入更明确的研发路线。

第一步是数据特性分析。我要先分析 Silesia 不同文件的字节分布、重复性、局部相关性和上下文规律。先知道数据为什么能被压缩，才能判断算法为什么有效。

第二步是 PAQ8PX 拆解。我会重点看上下文模型、模型混合、概率更新、Range Coding、文件类型识别和预处理。目标不是照搬整个 PAQ8PX，而是搞清楚高压缩率传统压缩器强在哪里。

第三步是 baseline 自动化。统一测试 gzip、zstd、LZMA/xz、PAQ8PX、cmix、NNCP/DeepZip，建立稳定结果表，记录版本、参数、硬件、BPB、速度、内存和 bit-exact 校验。

第四步是神经网络增强切入点。我不会一开始完全重写压缩器，而是先考虑把神经网络作为概率预测模块，替代或增强传统上下文模型、模型混合部分。这样改动更局部，也更容易做消融。

第五步是初步改进实验。选择一个局部模块，做可控增强，对比压缩率、速度和内存。如果收益很小但代价很大，就及时止损；如果某类数据上有稳定收益，再继续深入。

最后我想总结一下：我现在还没有进入“已经改出结果”的阶段，但我已经把资源入口、算法谱系、任务理解、实验框架和评估口径搭起来了。下一步就是把这个框架落到真实 Silesia baseline、PAQ8PX 拆解和局部模块改进上。这样后续无论结果是正向还是负向，都能形成可解释、可复查、可继续推进的研究路径。

# 版本 C：博士研究路线导向版

> 使用场景：适合导师更关心“你博士阶段到底要研究什么问题”的场合。这个版本更强调研究问题、技术主线和后续可形成论文/模块创新的方向。

老师，如果从博士研究路线角度看，我这阶段做的事情可以概括为：我在把无损压缩从一个算法集合，重新整理成一个围绕“数据冗余建模”的研究问题。

我现在的理解是，无损压缩的核心不是某个具体压缩器，而是三个层次的问题。第一层是数据层：不同数据中到底存在什么冗余，冗余以什么统计结构出现。第二层是模型层：压缩器怎样发现这些冗余，怎样预测下一个符号。第三层是编码层：概率模型怎样通过 Huffman、Arithmetic Coding、Range Coding、ANS 等熵编码后端变成实际 bitstream。后续如果我要做研究改进，也应该落在这三个层次之一，而不是泛泛地说“改进压缩算法”。

为了支撑这个研究路线，我先做了 Compression Research Atlas。这个网站对我不是资料展示，而是研究地图。它把理论基础、数据冗余、熵编码、PAQ/CMIX、神经压缩、领域专用压缩和实验复现组织到一起。当前项目里已经有核心文献入口、资源库、算法演进图谱、实验向导、Algorithm Board 和 Reading Paths。它的价值在于帮助我把“论文阅读”变成“问题定位”：一篇论文到底是在改概率模型、改预处理、改熵编码、改数据表示，还是改工程部署。

从算法演进看，我现在能把无损压缩理解成一条逐步增强概率预测能力的路线。Huffman 解决已知频率后的码长分配；LZ77/LZ78 把冗余利用从单符号频率扩展到重复片段；DEFLATE/gzip 把 LZ77 和 Huffman 做成成熟工程组合；BWT/bzip2 说明可逆变换可以让数据更容易被后端模型压缩；PPM 开始系统利用上下文概率；PAQ/PAQ8PX/cmix 通过大量上下文专家和模型混合追求极限预测精度；zstd/LZMA/brotli 则把算法能力放到工程权衡中；神经网络压缩进一步把预测器替换成可学习模型。

这条线可以总结为：频率统计 -> 重复匹配 -> 数据变换 -> 上下文预测 -> 模型混合 -> 神经网络概率建模。它给我的启发是，后续研究不一定要发明一个完全新的压缩器，而是可以在“预测能力”和“工程代价”之间找可验证的局部改进。

所以我的后续研究问题可以更具体地拆成几类。

第一个问题是数据特性分析。Silesia 里的不同文件为什么压缩难度不同？文本、代码、图像、医学数据、二进制数据分别有哪些可利用冗余？如果我不先分析数据特性，就很难解释某个压缩器为什么有效，也很难设计合理改进。

第二个问题是传统高压缩率模型拆解。PAQ8PX/cmix 这类压缩器很慢，但它们说明传统上下文建模还有很强能力。我要拆的是它们的上下文模型、模型混合、概率更新、文件类型识别和 Range Coding，而不是简单照搬整个系统。这里可能产生的研究价值是：哪些上下文专家真正贡献压缩率？哪些预处理对特定数据有效？哪些模型混合方式可以简化？

第三个问题是神经概率模型如何接入传统压缩器。我不打算一开始做端到端神经压缩器，因为那样工程量太大，也很难解释收益来源。更稳的路线是把神经网络作为一个概率预测模块，接到传统上下文模型或 mixer 里，再用统一 baseline 做消融。这样可以回答：神经模型到底补充了传统上下文模型的哪类冗余？收益是否抵得上速度和内存代价？

第四个问题是可复现评估。压缩算法研究很容易因为数据集、参数、版本、硬件不同而比较不公平。所以我会以 Silesia 为第一个统一 baseline，记录压缩器版本、命令、参数、时间、内存、BPB 和 SHA-256 校验。只有把 baseline 固定下来，后续模块改进才有可信度。

在这个研究视角下，我现在还没有宣称任何算法改进结果，但我已经把研究入口、算法谱系、实验框架和问题分解搭起来了。下一步我会先跑 Silesia baseline，再拆 PAQ8PX，再做局部模块增强实验。最终我希望形成的不是一个“调参式压缩器”，而是一个可解释的研究路径：从数据冗余分析出发，拆解强传统模型，再引入可控神经预测模块，并用严格 baseline 验证收益和代价。

# 版本 D：短时间口头汇报版

> 使用场景：适合只有 5-8 分钟，导师临时问“你最近做到哪了”的场合。这个版本更短，重点是讲清楚主线。

先说资源网站。项目现在叫“压缩算法研图 / Compression Research Atlas”。我对它的定位不是展示页，而是后续学习、复现和改进的工作台。里面已经有论文、数据集、压缩器项目、榜单、标准、术语、学习路线、实验向导和算法模块看板。比如资源库里有 Silesia、Canterbury、enwik、paq8px、zstd、brotli、CompressAI 等；算法演进里有 Huffman、LZ77、DEFLATE、BWT、PPM、PAQ、PAQ8PX、cmix、zstd、LZMA 和神经压缩；实验向导里也已经把数据集、压缩器、块大小、指标和结果记录拆出来了。

这个网站对我的意义是，它不是“我收集了很多资料”，而是让我后面能从看资料走向跑 baseline、拆模块和做改进。当然，目前真实本地实验还没有完全闭环，所以我不会说实验已经跑完，只能说框架已经搭好。

第二是任务理解。我现在理解无损压缩的本质，是在完全可逆还原的前提下，用更少 bit 表示原始数据。它压掉的是数据里的冗余，包括频率冗余、重复片段冗余、上下文规律冗余和结构化冗余。压缩器输入也不能简单说成字符，更准确地说是符号序列，最常见是 bit 或 byte 序列。不同数据能被压缩多少，取决于它里面有什么冗余，以及压缩器能不能预测这些冗余。

第三是算法演进。我现在把它理解成一条线：频率统计 -> 重复匹配 -> 数据变换 -> 上下文预测 -> 模型混合 -> 神经网络概率建模。Huffman 解决频率到码长，LZ77 利用重复片段，DEFLATE/gzip 是 LZ77 + Huffman 的工程组合，BWT/bzip2 代表先变换再压缩，PPM 强调上下文预测，PAQ/PAQ8PX/cmix 走高压缩率上下文混合路线，zstd/LZMA/brotli 是现代工程 baseline，NNCP/DeepZip/LLM compression 则是神经概率建模方向。

后续实验我准备从 Silesia corpus 做起。Silesia 不是为了刷榜，而是为了建立可复现 baseline。我会按技术路线测试 gzip、zstd、xz/LZMA、PAQ8PX、cmix、NNCP/DeepZip 等。指标不只看压缩率，还要看 BPB、压缩时间、解压时间、内存、吞吐和 SHA-256 round-trip 一致性。如果还没有本地实测数据，我会明确写成实验设计，不会编造结果。

最后我的阶段性判断是：gzip 是经典兼容 baseline，zstd 是现代高速工程 baseline，LZMA/xz 是高压缩率工程 baseline，PAQ8PX/cmix 是高压缩率研究对象，NNCP/DeepZip 是神经概率建模方向。后续我会先补 Silesia baseline，再拆 PAQ8PX，再考虑把神经网络作为概率预测模块接入传统压缩器，做局部增强实验。
