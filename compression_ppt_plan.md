## 第 6 页：无损压缩任务本质

**页面核心观点**

无损压缩是在完全可逆还原前提下，用更少 bit 表示原始数据。

**页面放什么**

- 必须完全还原
- 压缩的是冗余，不是“删除不重要信息”
- 冗余类型：频率冗余、重复片段冗余、上下文规律冗余、结构化冗余
- 上限由信息量、熵和可预测性决定

**建议图示**

数据冗余 -> 概率预测 -> 熵编码 -> 更短 bitstream

底部放一句：无损底线 = 解压后逐字节一致。

**我讲这一页时怎么说**

“我现在理解，无损压缩表面是在压文件，本质是在找数据里的冗余。只要解压不能逐字节还原，就不能叫严格无损。”

**注意不要怎么说**

不要说“去掉不重要信息”，那是有损压缩。

## 第 7 页：压缩器输入不是字符

**页面核心观点**

压缩器输入更准确地说是符号序列，最常见是 bit 或 byte 序列。

**页面放什么**

- 文本：人看到字符，压缩器看到 UTF-8/ASCII 字节
- 医学图像：人看到切片，通用压缩器看到 byte stream
- Silesia：文本、程序、图像、医学数据都能进入通用压缩器
- 不同数据的区别在于可利用冗余不同

**建议图示**

三列表：人眼看到 / 压缩器看到 / 可利用冗余。

**我讲这一页时怎么说**

“这里我修正了一个容易说错的地方。压缩器输入不是简单的字符，而是符号序列，最常见就是字节流。”

**注意不要怎么说**

不要说“压缩器输入一般是字符”。

**页面核心观点**

## 第 12 页：Huffman 到 LZ

**页面核心观点**

Huffman 解决频率到码长的问题，LZ 开始利用重复片段。

**页面放什么**

- Huffman：高频短码，低频长码
- 解决：已知频率后怎么编码
- LZ77：滑动窗口，引用历史片段
- LZ78：动态短语字典
- 转变：从单符号频率到重复片段复用

**建议图示**

左边 Huffman tree，右边 LZ77 窗口匹配。

**我讲这一页时怎么说**

“Huffman 更像是在回答概率到码长的问题，LZ 则开始利用序列里的重复片段。”

**注意不要怎么说**

不要说 LZ77 只是频率统计。

## 第 13 页：DEFLATE/gzip

**页面核心观点**

gzip 是经典兼容 baseline，核心是 LZ77 匹配加 Huffman 编码。

**页面放什么**

- DEFLATE = LZ77 matching + Huffman coding
- gzip = 常见文件封装和工具生态
- 优点：成熟、稳定、跨平台
- 局限：建模能力有限，压缩率不是最高
- 本研究定位：经典兜底参照

**建议图示**

字节流 -> LZ77 token -> Huffman -> gzip 文件

**我讲这一页时怎么说**

“gzip 对我来说重要，不是因为它最强，而是因为它最适合作为基础参照。”

**注意不要怎么说**

不要把 gzip 和 DEFLATE 完全等同。

## 第 14 页：BWT/bzip2

**页面核心观点**

BWT 体现“先变换数据结构，再压缩”的思路。

**页面放什么**

- BWT 不是完整压缩器
- 它通过重排让相似上下文聚集
- 后续再接 MTF、RLE、Huffman
- 启发：可逆预处理能降低后端建模难度

**建议图示**

原始序列 -> BWT 重排 -> 相似上下文聚集 -> 后端编码

**我讲这一页时怎么说**

“这条线给我的启发是，有时候不是编码器不够强，而是原始数据还没变成容易预测的形式。”

**注意不要怎么说**

不要说 BWT 本身就是完整压缩器。

## 第 15 页：PPM、PAQ、PAQ8PX、cmix

**页面核心观点**

高压缩率路线靠更强上下文预测和模型混合，但代价是速度和内存。

**页面放什么**

- PAQ：多个上下文专家在线混合
- PAQ8PX：高压缩率传统压缩器重点拆解对象
- cmix：更极限的 context mixing 路线
- 代价：慢、内存高、工程复杂

**建议图示**

## 第 21 页：实验指标

**页面核心观点**

无损压缩实验必须同时记录压缩率、速度、资源和正确性。

**页面放什么**

基础指标：

- 原始大小、压缩后大小
- compression ratio、BPB
- 压缩时间、解压时间
- 峰值内存
- SHA-256 round-trip 一致性

衍生指标：

- 压缩/解压吞吐 MB/s
- BPB 相对 gzip 节省
- 解压/压缩时间比
- 内存字节/原始字节
- 每 1% 提升付出的时间代价

场景指标：

- 32KB/64KB/128KB 延迟
- P50/P95/P99/max
- 多线程吞吐、冷热数据差异

**建议图示**
