# 压缩算法科研平台汇报讲稿

> 用途：这份文档是“对着网页直接讲”的汇报稿。  
> 建议：汇报时只念“可念稿”，把“演示动作”当作操作提示，不需要念出来。

## 0. 汇报开场

各位老师好，我这次汇报的是一个面向无损压缩算法研究的科研平台。它不是一个单纯的资料收藏网页，而是把“论文证据、SOTA 榜单、数据集基准、算法脉络、实验复现、任务管理和项目交付”放在同一个工作台里。

我设计这个平台的目标有三个：第一，帮助我系统学习无损压缩算法；第二，把论文、算法、数据集和实验任务组织成可以追踪的研究链路；第三，在后续写论文、做复现和阶段汇报时，能够快速拿出证据、引用、实验计划和交付材料。

所以我今天不会只介绍页面外观，而是按研究流程来讲：先看平台总览，再看证据库，然后看算法理解层，最后看实验和项目交付。

## 1. 平台总体定位

这个网站可以概括为“无损压缩算法研究工作台”。它服务的对象主要是三类：

1. 对我自己来说，它是学习和复现压缩算法的知识库。
2. 对课题组来说，它是整理文献、实验和研究进度的平台。
3. 对老师或项目验收方来说，它是一个可以展示研究过程、证据来源和阶段成果的汇报入口。

我把整个平台分成五层：

| 层级 | 对应页面 | 作用 |
| --- | --- | --- |
| 研究总览层 | Today、Project | 快速说明平台状态、研究进展和交付计划 |
| 证据资源层 | Library、SOTA、Datasets、Sources、Core Papers、Map | 管理论文、榜单、数据集、资源来源和研究证据 |
| 知识学习层 | Reading Paths、Tutorials、Terms、Standards、Notes | 支撑系统学习、术语理解、标准查询和笔记沉淀 |
| 算法理解层 | Algorithm Board、Evolution、Catalog | 解释算法模块、历史演化、算法结构和实现细节 |
| 执行交付层 | Experiments、Tasks、Calendar、Resource Library | 把研究对象转成实验、任务、计划和可交付材料 |

这样分类的原因是：压缩算法研究不能只看论文，也不能只看代码。一个算法是否值得研究，需要同时回答几个问题：它来自什么论文或标准、在哪个数据集上有效、属于哪条技术路线、能不能复现、能不能形成实验结果、最后能不能支撑论文或项目交付。

## 2. 推荐演示路线

如果从当前网页开始，我建议按照下面顺序演示：

1. 先打开 **Project / 项目交付总览**，说明这个平台服务于一个完整研究项目。
2. 再打开 **Today / 科研驾驶舱**，说明平台当前收录了哪些研究资产。
3. 进入 **Library / 文献库**，重点展示文献筛选、正式引用导出和资源区分。
4. 进入 **SOTA / 榜单** 和 **Datasets / 数据集**，说明榜单和数据集为什么拆开。
5. 进入 **Learned Compression / 深度学习压缩器**，说明它只负责学习式压缩方向，不替代总榜单。
6. 进入 **Reading Paths、Tutorials、Terms、Standards**，说明学习体系。
7. 进入 **Algorithm Board、Evolution、Catalog**，说明算法理解和技术脉络。
8. 进入 **Experiments、Tasks、Calendar**，说明如何落到实验复现和研究执行。
9. 最后回到 **Project / 交付面板**，收束到项目计划和后续工作。

## 3. Project / 项目交付总览

### 可念稿

这一页是项目交付总览，我把它放在系统层，是因为它面向的是老师或项目验收视角。它不只是告诉大家我做了哪些网页，而是把整个研究工作拆成阶段、交付物、风险和验收节点。

这里可以看到项目周期、阶段数量、关键交付物和验收节点。下面的阶段卡片把项目分成五个阶段：研究立项、算法拆解、方案设计、实验验证和成果移交。这个结构对应的是一个比较完整的科研项目过程：先确定研究边界和核心文献，再拆解主流算法，然后提出自己的方案，之后做实验验证，最后沉淀成论文、报告或演示系统。

中间的年度计划有多个视角，比如验收路线图、研发执行计划和论文产出计划。这样做的原因是，同一个研究项目从不同角色看重点不一样：验收方关心交付节点，研发方关心实验和实现，论文写作关心相关工作、方法、实验和投稿节奏。

交付物表格是这一页的重点，它把立项书、核心文献与引用库、算法拆解报告、baseline 复现实验包、自研模块接口规范、全量实验报告、演示站与知识库、最终验收包都列出来。每个交付物都有阶段、责任人、证据形式和验收标准。这样后续不会只说“我做了研究”，而是能说明每个阶段具体交付什么。

### 演示动作

打开 `/project-overview`，依次展示顶部概览、五个阶段卡片、年度计划切换、交付物登记表、风险和模板链接。

### 老师可能会问

**问题 1：为什么先讲交付面板，而不是先讲论文或算法？**  
回答思路：因为这个平台不是零散网页，而是服务于一个研究项目。先讲交付面板可以说明研究目标、阶段安排和验收标准，后面再讲文献、算法和实验时就有了上下文。

**问题 2：这些交付物是不是现在都已经完成了？**  
回答思路：不是全部完成。这里是项目交付结构和计划视图，已经完成的是网站框架、文献库、引用导出、部分算法档案和实验入口；后续要继续补充真实实验结果、算法结构图和正式报告材料。

**问题 3：项目计划为什么分成验收、研发和论文三个视角？**  
回答思路：因为科研项目同时面向管理、实现和论文产出。验收视角强调节点和材料，研发视角强调复现和实验，论文视角强调相关工作、方法和实验设计。

## 4. Today / 科研驾驶舱

### 可念稿

这一页是科研驾驶舱，相当于平台首页。它的作用是让第一次打开网站的人在第一屏就知道这个平台在做什么、当前有哪些资源、下一步应该进入哪里。

顶部指标展示的是平台当前的研究资产，包括文献数量、SOTA 榜单数量、深度学习压缩器条目、开源代码入口、笔记和任务。这个设计的目的不是追求数字好看，而是让研究状态可量化。

下面的“下一步研究入口”按研究判断链路组织：先核验 SOTA 榜单，再查看深度学习压缩器方向，然后选择数据集基准，接着回到文献库和实验台。也就是说，平台引导用户从“发现问题”走到“查证证据”，再走到“设计实验”。

右侧任务区显示当前执行状态。如果任务为空，它会给出从 Library、SOTA 和 Learned Compression 创建任务的入口，避免自动生成一堆没有证据来源的演示任务。

### 演示动作

打开 `/`，展示顶部指标、下一步研究入口、深度学习压缩器摘要、榜单核验状态和任务入口。

### 老师可能会问

**问题 1：首页的这些数字从哪里来？**  
回答思路：来自站点内部的数据源，比如文献库、SOTA 榜单数据、深度学习压缩器条目、笔记和任务状态。它不是手写宣传数字，而是和各模块联动的统计结果。

**问题 2：为什么首页不直接放很多论文？**  
回答思路：首页的职责是总览和导航，不适合堆细节。论文细节放在 Library，榜单放在 SOTA，算法细节放在 Algorithm Catalog，这样信息架构更清楚。

**问题 3：任务为空会不会显得平台没有内容？**  
回答思路：不会。任务为空说明当前没有预置演示任务。任务应该从具体论文、榜单或实验对象生成，这样任务才有证据来源，不会变成无意义待办。

## 5. Library / 文献、标准、代码与数据集证据库

### 可念稿

这一页是整个平台最核心的页面之一。我把它叫做事实数据库，而不是简单的文献卡片墙。原因是写论文和做实验时，我们需要区分不同类型的证据：正式论文可以进入参考文献，预印本需要谨慎使用，标准和官方文档更多用于方法或格式说明，代码仓库和数据集更多用于复现和附录支撑。

文献库顶部的统计区显示记录总数、论文数量、可正式引用数量、被排除的非正式资源以及预印本数量。这里特别重要的一点是：文献库默认不会把所有资源都混在一起导出引用。比如 arXiv-only 条目、未发表条目、代码仓库、数据集和标准文档，需要根据用途分别处理。

筛选区分成几类：第一是关键词搜索，可以按标题、作者、标签、摘要和章节匹配；第二是资源类型筛选，比如论文资料、标准文档、代码与项目、数据集或官方页面；第三是引用状态筛选，比如可引用、非文献、重复、预印本；第四是章节、年份、难度和优先级筛选。

文献库有两种视图：阅读视图适合逐篇浏览，批量管理视图适合选中多条记录做任务、笔记或引用导出。点击一条记录可以查看详情，包括来源链接、附件、标签、关联路线、阅读任务和引用操作。

我重点做了“一键导出引用库”。导出时可以选择当前筛选结果或手动选中的条目，格式支持 GB/T 7714 顺序编码、IEEE、APA 7th、BibTeX、RIS、CSL-JSON 和 Markdown 书目。默认策略是保守的：优先导出正式发表、元数据完整、没有重复的文献。预印本和非论文资源不会默认混入正文参考文献，除非手动打开相关选项。

这里还设计了去重逻辑。去重主要按 DOI、arXiv ID 和标题做判断。比如同一篇工作既有 arXiv 版本又有会议或期刊版本，正式写论文时应该优先使用正式版本的 DOI 和出版信息，而不是继续引用 arXiv 页面。

### 演示动作

打开 `/library`，先展示统计卡片，再切换资源类型和引用状态筛选。然后选择几篇正式文献，打开“导出引用库”，展示 GB/T 7714、IEEE、BibTeX 等格式，说明默认排除预印本和非文献资源。

### 老师可能会问

**问题 1：为什么文献库里还放代码、数据集和标准？**  
回答思路：因为研究证据不只来自论文。代码用于复现，数据集用于评测，标准用于格式和兼容性说明。但它们不会默认混入论文参考文献，而是作为支撑资源单独管理。

**问题 2：arXiv 是不是未发表？为什么不能随便引用？**  
回答思路：arXiv 是预印本平台，不等于正式发表。很多论文先上 arXiv，后面会有会议或期刊版本。如果已经正式发表，就应该升级为正式 DOI 和出版信息；如果只有 arXiv，则需要按导师或期刊要求决定是否引用，并明确它是 preprint。

**问题 3：导出的 GB/T 7714 能直接放进论文吗？**  
回答思路：它可以作为论文参考文献初稿使用，但最终还需要人工核对作者、题名、期刊会议、年份、卷期页码和 DOI。平台的价值是把可引用条目筛出来并统一格式，最终提交前仍要按学校模板校对。

**问题 4：怎么避免重复文献？**  
回答思路：平台会按 DOI、arXiv ID 和标题做重复候选识别。人工确认后，正式版本优先，预印本版本作为辅助记录或删除。

**问题 5：文献库和核心论文页有什么区别？**  
回答思路：Library 是全量证据库，Core Papers 是从里面筛出的重点阅读集合。一个是数据库，一个是精选阅读入口。

## 6. SOTA / 榜单真相源

### 可念稿

SOTA 页面回答的问题是：在某个任务、数据集和指标下，当前代表性方法是什么。它和文献库不同，文献库管理的是论文和资源本身，SOTA 管理的是“榜单事实”。

页面左侧是领域筛选，顶部是榜单数量、条目数量、近年更新和代码证据。主区域以 leaderboard 形式展示，每个榜单都标注数据集、指标、更新时间、来源类型以及论文或代码链接。

我把 SOTA 单独拆出来，是因为压缩算法领域很容易出现“某篇论文说自己效果好”，但没有统一数据集、没有统一指标、没有公开代码，就很难判断它是不是可靠的 SOTA。所以这个页面强调可追溯来源，例如 Mahoney、Hutter Prize、CLIC 等公开榜单或主榜入口。

### 演示动作

打开 `/sota`，按领域筛选一个榜单，展开条目，指一下 Dataset、Metric、Updated、Source、Paper、Code。

### 老师可能会问

**问题 1：SOTA 榜单和文献库为什么不合并？**  
回答思路：文献库回答“有哪些资料”，SOTA 回答“在某个评测口径下谁表现最好”。它们关注对象不同，合并会让引用管理和性能判断混在一起。

**问题 2：榜单是不是实时更新？**  
回答思路：当前主要是人工整理和公开来源快照，优先保证来源可靠和字段可核验。后续可以接入自动抓取，但自动更新必须配去重和人工复核，否则容易引入错误。

**问题 3：SOTA 一定代表最适合我的研究吗？**  
回答思路：不一定。SOTA 代表特定数据集和指标下的表现，实际研究还要考虑速度、内存、可复现性、许可证和应用场景。

## 7. Datasets / 数据集与基准库

### 可念稿

数据集页面回答的是“用什么数据来评价压缩算法”。我把它从 SOTA 中拆出来，是因为压缩性能高度依赖数据分布。一个方法在文本上好，不代表在图像、科学数组或基因组数据上也好。

这里的数据集按通用、文本、图像、视频、音频、科学数据、基因组和结构化数据分类。每个数据集记录规模、引用状态、授权风险、使用边界和来源链接。

页面还提供风险筛选。比如有些数据集授权清楚、引用风险低；有些需要确认版本、子集、使用许可或挑战赛规则。对于论文实验来说，数据集不是随便下载就能用，必须说明来源、版本、预处理方式和使用边界。

### 演示动作

打开 `/datasets`，切换领域和风险筛选，点击一个数据集详情，展示来源链接和“进入实验台”的入口。

### 老师可能会问

**问题 1：为什么要把数据集按领域分类？**  
回答思路：因为不同数据类型的冗余结构不同。文本有上下文和重复词，图像有二维局部相关，科学数组有数值连续性，基因组有字符序列和重复片段，所以评测不能混为一谈。

**问题 2：授权风险是什么意思？**  
回答思路：它提醒我在论文或项目中使用数据前，要确认许可、版本和引用要求，避免后续复现实验或公开材料出现合规问题。

**问题 3：数据集页面和实验台怎么连接？**  
回答思路：数据集详情可以进入实验台，实验台再选择压缩器、块大小和指标，把数据集变成可执行的实验配置。

## 8. Learned Compression / 深度学习压缩器库

### 可念稿

这一页的定位是深度学习压缩器库，专门收录深度学习实际参与压缩器设计的工作。它不替代总 SOTA 榜单，而是作为前沿方向雷达。

这里的分类按模态和技术对象组织，比如 LLM 无损压缩、神经图像 codec、神经视频 codec、神经音频 codec、神经熵模型、生成式低码率方向等。筛选条件包括年份范围、是否有代码、是否带 benchmark、许可证、成熟度和实现语言。

这样做的原因是，深度学习相关压缩工作非常分散。有些是完整 codec，有些只是熵模型，有些用于模型压缩，有些用于图像或视频压缩。如果全部放在一个普通文献列表里，很难判断它到底属于哪个方向、能不能复现、是否有公开 benchmark。

### 演示动作

打开 `/neural-hub`，切换 2024+、2025+、有代码、带 benchmark 和模态筛选，再在列表和卡片视图之间切换。

### 老师可能会问

**问题 1：为什么这个页面叫深度学习压缩器，而不是神经网络压缩？**  
回答思路：这里关注的是“深度学习参与数据压缩器设计”，比如神经熵模型、学习式 codec 和 LLM 压缩，不是单纯把神经网络模型本身变小的模型压缩。

**问题 2：这里的条目是不是都代表 SOTA？**  
回答思路：不是。这个页面是前沿雷达，记录新方向和可复现资源。是否 SOTA 还要回到 `/sota`，看具体数据集、指标和公开榜单证据。

**问题 3：为什么要看许可证和实现语言？**  
回答思路：因为后续复现实验和项目使用会受许可证、代码成熟度和实现语言影响。有代码但许可证不清，也不一定适合直接使用。

## 9. Research Map / 研究图谱

### 可念稿

研究图谱页面用于从多个视角查看文献结构。它不是逐篇阅读入口，而是帮助我看清文献分布和章节关系。

页面支持网格、列表、时间线、树状和径向视图。网格视图按章节聚合文献，时间线可以看年份分布，树状和径向视图可以看不同章节的占比。点击文献可以打开详情。

这个页面适合在汇报时说明“我的文献不是随便收集的，而是按研究章节和主题组织过的”。它也能帮助发现某些章节文献过少、某些年份覆盖不足的问题。

### 演示动作

打开 `/map`，切换网格、时间线、树状或径向视图，按章节筛选并点击一篇文献详情。

### 老师可能会问

**问题 1：研究图谱和文献库有什么区别？**  
回答思路：文献库用于筛选、导出和管理；研究图谱用于观察文献结构和主题分布。一个偏操作，一个偏理解全局。

**问题 2：图谱能说明研究是否全面吗？**  
回答思路：它不能自动证明全面，但能暴露覆盖情况，比如哪些章节和年份有空缺，帮助我继续补文献。

## 10. Core Papers / 核心论文

### 可念稿

核心论文页是从文献库中筛出的重点阅读集合。它按奠基、必读、选读和扩展分优先级，每篇都会说明为什么核心，以及读完能获得什么。

这样做的原因是，文献库条目很多，不可能一开始全部精读。核心论文页相当于研究入口，先保证关键算法、关键理论和关键综述能读透，再逐步扩展到细分方向。

### 演示动作

打开 `/core`，按优先级筛选，展示“为什么核心”和“读完能获得”的字段，然后演示加入阅读任务或笔记任务。

### 老师可能会问

**问题 1：核心论文怎么选出来？**  
回答思路：主要看它是否是奠基理论、主流算法来源、重要综述、标准或关键 benchmark 的来源，以及它是否支撑后续实验和论文写作。

**问题 2：核心论文数量会不会太少？**  
回答思路：核心集合是优先阅读列表，不代表全部文献。全部资源在 Library 中，核心论文会随着研究推进继续调整。

## 11. Reading Paths / 阅读路线

### 可念稿

阅读路线页是把论文组织成学习路径，而不是简单按年份或标题排列。每条路线都有研究目标、适用对象、产出标准、能力目标、论文顺序和实验产出。

比如我可以有一条基础路线，用来理解信息论、熵编码和字典压缩；也可以有一条工程路线，围绕 zstd、brotli、PAQ、CMIX 等压缩器拆解；还可以有深度学习压缩路线，围绕神经熵模型和学习式 codec。

每条路线可以加入任务或排到日历。这样它不只是“我应该读哪些论文”，而是能变成可执行的阅读计划。

### 演示动作

打开 `/reading-paths`，切换左侧路线，展示研究目标、适用对象、产出标准、论文时间线、实验与证据产出，并演示“整条入任务”。

### 老师可能会问

**问题 1：为什么需要阅读路线，不直接按文献年份读？**  
回答思路：年份只能说明历史顺序，不能说明学习目标。阅读路线按能力目标组织，更适合从基础到复现再到论文产出。

**问题 2：路线完成的标准是什么？**  
回答思路：不是读完就算完成，而是要形成笔记、算法理解、实验设计或综述材料，这些产出标准会写在路线里。

## 12. Tutorials / 教程资源

### 可念稿

教程页不是自己写几篇入门文章，而是作为教程资源导航。它把公开教程、官方文档、课程、视频、benchmark 指南、代码教程和论文导读按照类别管理。

分类包括基础教程、算法拆解、数据说明、benchmark、实现教程、标准规范、前沿教程和课程资源。页面支持封面卡片视图和矩阵列表视图，也能按媒体类型、难度、来源质量和是否视频筛选。

我这样设计是因为不同学习阶段需要不同资源：刚入门要看信息论和压缩管线，做实验要看数据集和 benchmark，拆代码要看实现教程，追前沿要看论文导读和代码库。

### 演示动作

打开 `/tutorials`，切换分类、视频筛选和卡片/矩阵视图，点击一个资源展示外部链接。

### 老师可能会问

**问题 1：教程资源和阅读路线有什么区别？**  
回答思路：阅读路线以论文为主，目标是研究能力；教程资源以外部学习材料为主，目标是快速补背景、查实现和理解工具。

**问题 2：教程资源质量怎么保证？**  
回答思路：页面记录来源质量，比如官方、标准、大学课程、benchmark、作者资源和社区资源。正式引用和论文依据仍以 Library 和 SOTA 为准。

## 13. Terms / 术语库

### 可念稿

术语库用于解决压缩算法学习中的概念问题。压缩领域有很多容易混淆的词，比如 entropy、arithmetic coding、range coding、context modeling、bitstream、bpp、BPB、lossless、near-lossless 等。

这里按基础概念、信息论、熵编码、字典、变换、学习式压缩、指标、标准和领域场景分类。每个术语都有中文解释、外部百科链接和关联论文。

术语库的价值在于汇报和写论文时统一概念口径，避免同一个词在不同页面解释不一致。

### 演示动作

打开 `/terms`，搜索一个术语，切换分类和排序，展示关联论文入口。

### 老师可能会问

**问题 1：术语解释是否可以作为论文引用？**  
回答思路：术语页主要是学习和口径统一，不直接作为论文参考文献。论文引用仍然使用正式论文、标准或教材。

**问题 2：为什么术语要关联论文？**  
回答思路：这样不是停留在词典解释，而是能回到具体算法或论文中理解它的作用。

## 14. Standards / 标准与场景

### 可念稿

标准页按使用场景组织标准、格式和 benchmark。比如 Web 文本、图像、医学影像、天文数据、科学数组、表格数据和评测基准，它们对应的压缩标准和使用边界不同。

每个场景卡片会说明为什么重要、适用数据形态、相关标准、什么时候需要看、官方链接和相关文献。这样可以帮助我在做实验或写论文时明确：我研究的是通用无损压缩，还是特定格式、特定行业数据或特定 benchmark。

### 演示动作

打开 `/standards`，点击一个场景，展示标准列表、适用数据、官方链接和相关文献。

### 老师可能会问

**问题 1：标准页和数据集页有什么区别？**  
回答思路：数据集页回答“用什么数据评测”，标准页回答“这个场景有哪些格式、规范和使用边界”。二者都服务实验，但侧重点不同。

**问题 2：为什么要关心标准？**  
回答思路：很多压缩方法不仅是算法问题，还涉及格式兼容、解码要求、行业规范和 benchmark 规则。不理解标准，实验结果可能无法和已有系统对齐。

## 15. Notes / 研究笔记

### 可念稿

研究笔记页是把阅读过程沉淀下来。它支持按文献创建笔记，也支持自由笔记。每篇笔记有状态、标签、字数、结构化摘要、关键词、摘录和六问模板。

六问模板包括：这篇论文要解决什么问题、用了什么方法、实验怎么做、结论是什么、局限性是什么、对我的启发是什么。这样能避免读完论文只记标题，不知道如何转化成综述和实验设计。

笔记可以导出 Markdown 或纯文本，也可以复制全文。后续写论文相关工作或准备组会汇报时，笔记就是材料来源。

### 演示动作

打开 `/notes`，展示笔记列表、状态筛选、六问模板、摘录、高亮、导出和复制功能。

### 老师可能会问

**问题 1：笔记和文献库有什么关系？**  
回答思路：文献库管理事实和元数据，笔记管理我的理解和产出。文献可以没有笔记，但重点论文应该逐步沉淀笔记。

**问题 2：六问模板是不是太固定？**  
回答思路：它是最低限度的结构化阅读框架，不限制自由写作。复杂论文还可以补公式、图、实验表和个人批注。

## 16. Algorithm Board / 算法模块板

### 可念稿

算法模块页的定位是“压缩器架构设计板”。它不是算法排行榜，也不是历史图谱，而是把一个完整压缩器拆成可替换的工程模块。

页面按输入识别、领域适配、字典匹配、变换预测、概率建模、神经模块、融合策略、熵编码、容器和评测闭环等职责组织。每个模块说明它解决的问题、输入输出、状态、难度、风险、影响指标、替代方案、关联文献和实验入口。

这个页面主要回答：如果我要设计或改进一个压缩器，应该在哪个模块动手？这个模块的输入是什么，输出是什么，会影响压缩率还是速度，复现风险在哪里？

### 演示动作

打开 `/algorithm-board`，先看流程图，再点开一个模块详情，展示输入输出、风险、关联文献和加入任务。

### 老师可能会问

**问题 1：算法模块页和算法目录有什么区别？**  
回答思路：算法模块页按压缩器内部职责组织，关注模块设计；算法目录按具体算法条目组织，关注某个算法的结构、流程、来源和实现细节。

**问题 2：为什么要把压缩器拆成模块？**  
回答思路：因为改进压缩器通常不是从零写一个算法，而是在预处理、建模、匹配、熵编码或参数策略等模块上改。模块化有助于复现和消融实验。

**问题 3：模块状态是什么意思？**  
回答思路：它表示当前是否只是规范设想、已有原型、可运行、已验证或可交付。这样能区分研究设想和真实可执行内容。

## 17. Algorithm Evolution / 算法演化图谱

### 可念稿

算法演化图谱用于解释压缩算法的历史脉络。它的重点不是把算法按卡片分类，而是说明不同算法之间的继承关系、改进动力和技术分支。

历史主线视图把年份放在横向，把技术路线放在纵向，通过节点和连线展示从信息论、Huffman、Arithmetic Coding、LZ 系列、DEFLATE、BWT、PPM、PAQ、Zstandard、Brotli，再到学习式压缩的演化路径。

技术分支视图更关注算法族谱，例如字典压缩路线、上下文建模路线、熵编码路线、工程压缩器路线和学习式路线。架构位置视图说明这些算法在压缩器流水线中的职责。性能取舍视图用于比较速度、压缩率和复杂度，场景适配视图用于说明不同算法适合哪些数据类型。

这一页适合解释“为什么会出现某个算法”。比如 LZ 系列解决重复字符串匹配问题，DEFLATE 把 LZ77 和 Huffman 结合成工程格式，PAQ 类方法追求极限压缩率但牺牲速度，Zstd 更强调工程平衡和部署。

### 演示动作

打开 `/algorithm-evolution`，先看历史主线，点击全屏，切换核心/全部范围，再切换技术分支、架构位置、性能取舍和场景适配。

### 老师可能会问

**问题 1：演化图谱是不是完整覆盖所有压缩算法？**  
回答思路：它不是百科全书式穷举，而是覆盖无损压缩主干路线和研究相关节点。后续可以继续补充专用压缩器、行业格式和最新学习式方法。

**问题 2：连线代表什么？**  
回答思路：连线代表技术影响或改进关系，比如思想继承、模块组合、工程标准化或性能取舍上的改进，不一定表示代码直接继承。

**问题 3：为什么需要多种视图？**  
回答思路：历史视图看时间，分支视图看算法族，架构视图看模块位置，取舍视图看性能定位，场景视图看适用数据。不同问题需要不同视角。

## 18. Algorithm Catalog / 算法档案库

### 可念稿

算法目录是具体算法的档案库。每个卡片对应一个算法、格式、压缩器或模型，记录出现年份、技术族、模块位置、来源数量、是否有公开表现、是否有结构图位和速度/压缩率定位。

点击卡片后，详情会展开几个部分：结构定位、技术价值、算法结构、压缩流程、算法实现、技术步骤拆解、来源与证据、结构图位、公开表现、适用场景和边界。

这里我特别强调“压缩流程”和“算法实现”。压缩流程不是一句话概括，而是从输入数据开始，说明数据如何分块、如何建模、如何预测或匹配、如何转成概率或符号，最后如何经过熵编码和容器封装。算法实现则关注实际写代码时需要的数据结构、状态维护、参数、解码一致性和复杂度。

如果公开 benchmark 有真实数值，就展示表格；如果没有可靠来源，就不硬填 NaN 或编造结果，而是保留结构图位置，后续补图和补实验。

### 演示动作

打开 `/algorithm-catalog`，按时间、分类、压缩率或速度排序，选择一个算法卡片，展示结构、流程、实现、来源与证据。

### 老师可能会问

**问题 1：算法目录里的性能分数是不是精确 benchmark？**  
回答思路：卡片上的速度和压缩率更多是定位和相对判断；只有详情里的公开表现表格才表示有来源的具体数值。没有可靠数据不会编造。

**问题 2：为什么有的算法没有结构图？**  
回答思路：当前先保留结构图位，后续会为重点算法补手绘或论文风格结构图。没有可靠图时不随便放网上不确定来源的图片。

**问题 3：算法实现部分为什么重要？**  
回答思路：论文理解到算法复现之间差距很大。实现部分能提醒关键数据结构、同步状态、解码一致性和复杂度风险，便于后续写代码和做实验。

## 19. Experiments / 实验复现

### 可念稿

实验台把前面的论文、数据集和算法变成可执行的实验配置。它采用六步向导：选择数据集、选择 baseline、选择块大小、选择指标、生成命令、记录结果。

这里强调无损压缩实验的几个关键点：第一，必须做 bit-exact 验证，也就是压缩后再解压必须和原文件逐字节一致；第二，评估不能只看压缩率，还要记录吞吐、内存、块大小、硬件、参数和版本；第三，实验要能拆成任务，比如准备数据、跑实验和整理报告。

实验台不是替代真实终端运行，而是帮助规范实验设计和记录。后续真正跑命令后，可以把 BPB、压缩率、速度和 bit-exact 状态回填。

### 演示动作

打开 `/experiments`，从 Step 1 到 Step 6 走一遍，展示数据集、baseline、块大小、指标、命令模板和创建任务。

### 老师可能会问

**问题 1：为什么要记录块大小？**  
回答思路：块大小会影响压缩器能看到的上下文范围、内存占用和速度。不同数据集用不同块大小，结果不可直接比较，所以必须记录。

**问题 2：BPB 和压缩率有什么区别？**  
回答思路：BPB 是每字节平均使用的比特数，越低越好；压缩率通常表示压缩后相对原文件减少多少。论文中需要明确指标定义。

**问题 3：实验台能直接运行实验吗？**  
回答思路：当前主要是实验设计和任务生成入口，命令需要在本地环境执行。后续可以继续接自动运行、日志上传和结果表格。

## 20. Tasks / 研究任务

### 可念稿

任务页是执行层，用来把阅读、笔记、下载、实验和汇报拆成可追踪任务。它采用看板结构，包括待开始、进行中、等待材料、已完成和归档。

每张任务卡可以关联文献、阅读路线、算法模块或实验对象，并记录截止日期、预计时间、优先级和下一步动作。这个设计的意义是把“我需要研究某个东西”转成具体可推进的任务。

如果任务栏为空，页面会提供从 Library、SOTA、Learned Compression 和 Experiments 创建任务的入口。这样任务不是凭空生成，而是从证据对象产生。

### 演示动作

打开 `/tasks`，展示空状态或看板列，点击新建任务，说明任务类型、关联对象、状态和拖拽更新。

### 老师可能会问

**问题 1：任务页和日程页有什么区别？**  
回答思路：任务页管理事情本身，比如读论文、跑实验、写笔记；日程页管理时间安排，比如什么时候做、持续多久、是否冲突。

**问题 2：任务为什么要关联文献或算法？**  
回答思路：这样每个任务都有来源，后续回看时知道它是为了哪篇论文、哪个算法或哪个实验服务的。

## 21. Calendar / 日程

### 可念稿

日程页用于把研究任务放到时间维度。它适合安排论文阅读、实验运行、组会准备、阶段汇报和交付节点。

相比任务看板，日程更关注时间冲突和节奏控制。比如某一周如果同时安排大量论文阅读和实验复现，日程视图就能提醒工作量是否过载。

### 演示动作

打开 `/calendar`，展示事件列表、详情、统计和任务关联。

### 老师可能会问

**问题 1：日程是否能导出到外部日历？**  
回答思路：当前项目里已有 iCal 相关工具基础，后续可以继续完善导出到外部日历的能力。

**问题 2：日程和项目交付节点怎么关联？**  
回答思路：交付面板给出阶段目标，日程页把阶段目标拆成具体时间安排。

## 22. Sources / 研究动态与来源监控

### 可念稿

Sources 页面用于记录研究来源和动态入口。它不是最终证据库，而是发现新资料的入口。比如论文更新、榜单变化、数据集或官方项目页面，可以先进入 Sources，再经过人工筛选进入 Library、SOTA 或 Tutorial。

我对这个页面的定位比较谨慎：如果没有稳定联网更新，就不能把它包装成实时新闻系统。更合理的说法是来源监控和候选资料池，正式文献和榜单仍然要回到 Library 和 SOTA 复核。

### 演示动作

打开 `/research-feed`，说明这是候选来源监控，不把未核验内容直接当作正式证据。

### 老师可能会问

**问题 1：这个页面能自动联网更新吗？**  
回答思路：当前更偏来源管理和候选池。自动联网抓取可以做，但必须配去重、错误处理和人工复核，否则容易出现重复、过时或无关内容。

**问题 2：Sources 和 Library 的边界是什么？**  
回答思路：Sources 是发现入口，Library 是已整理的证据库。Sources 里的内容不一定可信，进入 Library 后才作为研究证据管理。

## 23. Resource Library / 资源库

### 可念稿

资源库用于收集不属于 SOTA 和数据集的其他研究资源，比如工具、框架、预训练或实现资源、教程、项目、标准和模板。

我把资源库和 SOTA、Datasets 拆开，是为了避免三类信息混乱：SOTA 是结果排名，Datasets 是评测基础，Resource Library 是工具和辅助资源。比如一个代码库、一个报告模板、一个课程或一个标准文档，不应该和榜单结果混在同一张表里。

### 演示动作

打开 `/hub`，按资源类型筛选，展示工具、课程、项目、标准和模板资源，说明它们可关联文献。

### 老师可能会问

**问题 1：资源库和教程页是否重复？**  
回答思路：教程页专门面向学习材料，资源库范围更广，包括工具、框架、项目、标准和模板。教程可以作为资源的一类，但资源库不只放教程。

**问题 2：资源库里的链接怎么保证有效？**  
回答思路：当前会记录来源链接和相关文献，后续可以增加链接状态检查和更新时间字段。

## 24. Settings / 设置

### 可念稿

设置页主要用于平台运行和个性化配置，比如主题、界面偏好或后续数据源配置。它不是汇报重点，但说明这个系统不是静态页面，而是为长期维护预留了配置入口。

### 老师可能会问

**问题 1：平台以后扩展时设置页能做什么？**  
回答思路：可以扩展数据源配置、引用格式偏好、主题、导出路径、外部工具集成，比如 Zotero、GitHub、实验日志目录等。

## 25. 重点汇报：为什么这样分类

如果老师问“为什么页面这么多”，可以这样回答：

这个平台的分类不是按网页数量划分，而是按研究链路划分。

第一类是证据判断：文献库解决“资料是什么”，SOTA 解决“结果是否领先”，数据集解决“评测依据是什么”，标准解决“使用边界是什么”。

第二类是知识理解：阅读路线解决“我应该按什么顺序学习”，教程解决“遇到背景问题去哪补”，术语库解决“概念口径是否统一”，笔记解决“我的理解是否沉淀下来”。

第三类是算法研究：算法模块解决“压缩器由哪些部分组成”，演化图谱解决“算法为什么这样发展”，算法目录解决“单个算法怎么工作、怎么实现、来自哪里”。

第四类是执行交付：实验台解决“如何设计复现实验”，任务和日程解决“如何推进”，项目交付面板解决“如何验收和汇报”。

所以它不是资料站，而是一个从资料、理解、复现到交付的研究工作流。

## 26. 常见总问题与回答

**问题 1：这个网站现在最核心的价值是什么？**  
回答：核心价值是把压缩算法研究中的证据、知识、算法和执行统一起来。它能帮我从一篇论文追到引用、数据集、SOTA、算法结构、实验任务和最终汇报材料。

**问题 2：目前最大的不足是什么？**  
回答：目前主要不足是部分内容仍需要继续补全真实实验结果、重点算法结构图和更严格的文献元数据校对。平台框架已经搭好，后续重点是把真实复现结果和论文级材料填进去。

**问题 3：它和 Zotero、Notion 这种工具有什么区别？**  
回答：Zotero 偏文献管理，Notion 偏通用笔记。这个平台是围绕无损压缩算法定制的，把 SOTA、数据集、算法图谱、实验台和交付面板都连起来，更贴近我的研究流程。

**问题 4：如果以后换研究方向，这个平台还能用吗？**  
回答：底层结构可以复用，因为“文献库、榜单、数据集、术语、算法目录、实验、任务、交付”是通用科研流程。需要替换的是领域数据和分类体系。

**问题 5：怎么保证内容不是 AI 编的？**  
回答：正式证据都应该回到来源链接、DOI、论文、标准、代码仓库和公开榜单。平台本身不把未核验内容当作最终结论，而是通过 Library、SOTA、Datasets 和 Sources 区分证据级别。

**问题 6：后续最应该优先补什么？**  
回答：第一是补正式可引用的顶会顶刊和标准文献；第二是给重点算法补结构图和实现细节；第三是跑 baseline 实验并回填结果；第四是把引用库和论文写作流程进一步打通。

## 27. 汇报收尾

最后总结一下，这个平台的核心目标不是做一个好看的网页，而是把无损压缩算法研究变成一个可持续维护的工作台。它把论文、榜单、数据集、算法、实验、任务和交付放在同一套结构里。

后续我会重点做三件事：第一，继续补充正式发表、可引用的高质量文献；第二，完善重点算法的结构图、流程拆解和实现说明；第三，围绕典型数据集跑 baseline 和改进实验，把平台从“研究管理工具”进一步推进到“实验与论文产出工具”。
